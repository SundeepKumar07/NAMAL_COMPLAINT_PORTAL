% 1. INTRODUCTION
\section{Introduction}

\subsection{Purpose}
This Software Requirements Specification (SRS) document provides a complete description of the Namal Complaint Management System (NCMMS). The document specifies all functional and non-functional requirements for the system and is intended for:

\begin{itemize}
    \item Development team members who will design and implement the system
    \item Project managers who will track project progress
    \item Testers who will verify and validate the system
    \item Requirement Provider (university administration) who will review and approve requirements
    \item Maintenance personnel who will maintain the system after deployment
\end{itemize}

\subsection{Scope}
The Namal Complaint Management and Maintenance System (NCMMS) is a web-based application designed to streamline the maintenance complaint management process at Namal University. The system will:

\begin{itemize}
    \item Replace the current informal, verbal complaint reporting system
    \item Provide a centralized platform for complaint submission and tracking
    \item Enable efficient assignment and monitoring of maintenance tasks
    \item Improve transparency and accountability in complaint resolution
    \item Generate reports and analytics on maintenance performance
\end{itemize}

\textbf{Benefits:}
\begin{itemize}
    \item Reduced complaint resolution time through systematic tracking
    \item Enhanced user satisfaction through real-time status updates
    \item Improved resource allocation and workload management
    \item Data-driven insights for maintenance planning
    \item Complete audit trail of all maintenance activities
\end{itemize}

\textbf{System will:}
\begin{itemize}
    \item Allow users (students, faculty, staff) to submit categorized complaints
    \item Enable administrators to assign complaints to maintenance staff
    \item Provide real-time complaint status tracking
    \item Generate summary reports and performance metrics
    \item Send notifications for complaint updates
\end{itemize}

\textbf{System will not:}
\begin{itemize}
    \item Handle financial transactions or budget management
    \item Manage inventory of maintenance materials
    \item Replace face-to-face emergency reporting procedures
    \item Provide external vendor management capabilities
\end{itemize}

% 2. OVERALL DESCRIPTION
\section{Overall Description}

\subsection{Product Perspective}
The Namal Complaint Management System (NCMMS) is a new, self-contained web-based application designed specifically for Namal University. It operates independently and does not integrate with existing university systems in its initial release.

\subsubsection{System Interfaces}
The system consists of the following major components:

\begin{itemize}
    \item \textbf{Web Application Interface} - Browser-based user interface accessible via standard web browsers
    \item \textbf{Backend Server} - Node.js/Express.js server handling business logic
    \item \textbf{Database System} - MongoDB for data persistence
    \item \textbf{Cloud Storage} - Cloudinary for image/file storage
    \item \textbf{Authentication Module} - JWT-based security system
\end{itemize}

\subsubsection{User Interfaces}
The system will provide responsive web interfaces optimized for:
\begin{itemize}
    \item Desktop browsers (Chrome, Firefox, Safari, Edge)
    \item Tablet devices
    \item Mobile devices (smartphones)
\end{itemize}

Key interface components include:
\begin{itemize}
    \item Login/Registration screens
    \item User dashboard
    \item Complaint submission form with image upload capability
    \item Complaint tracking interface
    \item Administrative panel for complaint management
    \item Maintenance staff work queue
    \item Reporting and analytics dashboard
\end{itemize}

\subsubsection{Hardware Interfaces}
\begin{itemize}
    \item \textbf{Client Side:} Any device with a modern web browser and internet connection
    \item \textbf{Server Side:} Web server capable of running Node.js applications
    \item \textbf{Storage:} Cloud storage for images and attachments via Cloudinary
\end{itemize}

\subsubsection{Software Interfaces}
\begin{itemize}
    \item \textbf{Operating System:} Platform-independent (runs on any OS supporting Node.js)
    \item \textbf{Database:} MongoDB version 5.0 or higher
    \item \textbf{Web Framework:} React/Next.js for frontend, Express.js for backend
    \item \textbf{Runtime Environment:} Node.js version 18.x or higher
    \item \textbf{Cloud Services:} Cloudinary API for media management
\end{itemize}

\subsubsection{Communications Interfaces}
\begin{itemize}
    \item HTTP/HTTPS protocols for client-server communication
    \item RESTful API architecture
    \item JSON data format for API requests and responses
    \item WebSocket connection for real-time notifications (optional)
    \item Email notifications via SMTP
\end{itemize}

\subsubsection{Memory Constraints}
\begin{itemize}
    \item Maximum file upload size: 5 MB per attachment
    \item Server memory: Minimum 2 GB RAM recommended
    \item Database storage: Scalable based on complaint volume
\end{itemize}

\subsection{Product Functions}
The NCMMS provides the following major functions:

\begin{enumerate}
    \item \textbf{User Management}
    \begin{itemize}
        \item User registration and authentication
        \item Role-based access control (Student/Faculty/Staff, Maintenance Staff, Administrator)
        \item Profile management
    \end{itemize}
    
    \item \textbf{Complaint Management}
    \begin{itemize}
        \item Submit new complaints with category, description, location, and images
        \item View and track complaint status
        \item Edit or cancel pending complaints
        \item Add comments or updates to existing complaints
    \end{itemize}
    
    \item \textbf{Administrative Functions}
    \begin{itemize}
        \item View all complaints with filtering and search capabilities
        \item Assign complaints to maintenance staff
        \item Update complaint status and priority
        \item Review and approve complaint resolution
        \item Manage user accounts and roles
    \end{itemize}
    
    \item \textbf{Maintenance Staff Functions}
    \begin{itemize}
        \item View assigned complaints
        \item Update work progress and status
        \item Upload completion photos or documentation
        \item Request reassignment if necessary
    \end{itemize}
    
    \item \textbf{Reporting and Analytics}
    \begin{itemize}
        \item Generate complaint resolution reports
        \item View performance metrics by category and staff member
        \item Export data in CSV format
        \item Display statistical dashboards
    \end{itemize}
    
    \item \textbf{Notification System}
    \begin{itemize}
        \item Email notifications for complaint status changes
        \item In-app notifications
        \item Reminder notifications for pending complaints
    \end{itemize}
\end{enumerate}

    \item \textbf{Technical Expertise:} Basic computer and internet skills

\begin{itemize}
    \item \textbf{Education Level:} Varies (technical school to undergraduate)
    \item \textbf{Technical Expertise:} Basic smartphone/computer operation skills
    \item \textbf{Usage Pattern:} Regular daily users to check assigned tasks
    \item \textbf{Expectations:} Clear work queue, easy status updates, mobile-friendly interface
\end{itemize}

\subsubsection{Administrators/Supervisors}
\begin{itemize}
    \item \textbf{Education Level:} Undergraduate or higher
    \item \textbf{Technical Expertise:} Intermediate computer skills, familiar with management systems
    \item \textbf{Usage Pattern:} Daily users for monitoring and management
    \item \textbf{Expectations:} Comprehensive oversight tools, reporting capabilities, user management
\end{itemize}

\subsection{Constraints}

\subsubsection{Regulatory Policies}
\begin{itemize}

    \item User data must be handled according to privacy regulations
    \item Authentication must follow university security standards
\end{itemize}

\subsubsection{Hardware Limitations}
\begin{itemize}
    \item Must function on standard university network infrastructure
    \item Should work on devices commonly available to students and staff
    \item Image uploads limited by available storage capacity
\end{itemize}

\subsubsection{Interface Requirements}
\begin{itemize}
    \item Must be accessible via standard web browsers
    \item Should not require plugin installations
    \item Must work within university firewall restrictions
\end{itemize}

\subsubsection{Development Constraints}
\begin{itemize}
    \item Development must follow Kanban methodology
    \item Must use MERN stack as specified

    \item Budget: Student project with limited resources
\end{itemize}

\subsubsection{Security Constraints}
\begin{itemize}
    \item All passwords must be encrypted

\subsubsection{Assumptions}

\subsubsection{Communications Interfaces}

\paragraph{COM-1: HTTP/HTTPS Protocol}
\begin{itemize}
    \item The system shall use HTTPS protocol for all client-server communication
    \item SSL/TLS encryption shall be enforced for production environment
    \item HTTP methods supported: GET, POST, PUT, PATCH, DELETE
\end{itemize}

\paragraph{COM-2: RESTful API}
\begin{itemize}
    \item API Architecture: REST (Representational State Transfer)
    \item Data Format: JSON for all request and response payloads
    \item Authentication: Bearer token (JWT) in Authorization header
    \item Base URL format: https://[domain]/api/v1/[resource]
\end{itemize}

\paragraph{COM-3: Email Communication}
\begin{itemize}
    \item Protocol: SMTP (Simple Mail Transfer Protocol)
    \item Port: 587 (TLS) or 465 (SSL)
    \item Email format: HTML with plain text fallback
    \item Content: Notification messages about complaint status changes
\end{itemize}

\paragraph{COM-4: WebSocket (Optional)}
\begin{itemize}
    \item Protocol: WebSocket for real-time notifications
    \item Library: Socket.io
    \item Purpose: Push instant updates to connected clients
    \item Fallback: HTTP polling if WebSocket not available
\end{itemize}

\subsection{Functional Requirements}

\subsubsection{User Authentication and Authorization}

\paragraph{FR-1.1: User Registration}
The system shall allow new users to register by providing:
\begin{itemize}
    \item Full name (mandatory)
    \item University ID number (mandatory)
    \item University email address (mandatory)
    \item Phone number (mandatory)
    \item Password meeting security criteria (mandatory)
    \item User role selection: Student/Faculty/Staff (mandatory)
    \item Department/Program (mandatory)
\end{itemize}

\paragraph{FR-1.2: Email Verification}
The system shall send a verification email to the registered email address and require users to verify their email before accessing the system.

\paragraph{FR-1.3: User Login}
The system shall authenticate users using email and password credentials and issue a JWT token upon successful authentication.

\paragraph{FR-1.4: Password Requirements}
The system shall enforce password requirements:
\begin{itemize}
    \item Minimum 8 characters
    \item At least one uppercase letter
    \item At least one lowercase letter
    \item At least one number
    \item At least one special character
\end{itemize}

\paragraph{FR-1.5: Password Reset}
The system shall allow users to reset forgotten passwords by:
\begin{itemize}
    \item Requesting password reset via email
    \item Receiving a time-limited reset link (valid for 1 hour)
    \item Creating a new password meeting security criteria
\end{itemize}

\paragraph{FR-1.6: Session Management}
The system shall maintain user sessions using JWT tokens with:
\begin{itemize}
    \item Token expiration time of 24 hours
    \item Automatic logout upon token expiration
    \item Refresh token capability for extended sessions
\end{itemize}

\paragraph{FR-1.7: Role-Based Access Control}
The system shall implement three user roles with distinct permissions:
\begin{itemize}
    \item \textbf{End User:} Submit complaints, view own complaints, update profile
    \item \textbf{Maintenance Staff:} View assigned complaints, update status, upload completion photos
    \item \textbf{Administrator:} Full access to all features including user management, complaint assignment, and reporting
\end{itemize}

\subsubsection{Complaint Management}

\paragraph{FR-2.1: Submit Complaint}
The system shall allow authenticated end users to submit a new complaint with:
\begin{itemize}
    \item Category selection from: Electrical, Plumbing, IT, Furniture, Civil, HVAC, Other (mandatory)
    \item Priority selection: Low, Medium, High, Critical (mandatory)
    \item Location: Building name and room number (mandatory)
    \item Title: Brief summary (max 100 characters) (mandatory)
    \item Description: Detailed explanation (max 1000 characters) (mandatory)
    \item Image attachments: Up to 3 images, max 5MB each (optional)
\end{itemize}

\paragraph{FR-2.2: Generate Ticket ID}
The system shall automatically generate a unique ticket ID for each complaint in the format: NCMMS-YYYY-NNNN (e.g., NCMMS-2025-0001).

\paragraph{FR-2.3: Complaint Validation}
The system shall validate complaint submissions:
\begin{itemize}
    \item All mandatory fields must be filled
    \item Images must be in supported formats (JPG, PNG, GIF)
    \item Image file size must not exceed 5MB per file
    \item Description must not contain inappropriate content
\end{itemize}

\paragraph{FR-2.4: Initial Complaint Status}
The system shall set the status of newly submitted complaints to "Open".

\paragraph{FR-2.5: View Own Complaints}
The system shall allow users to view a list of all their submitted complaints with:
\begin{itemize}
    \item Ticket ID
    \item Title
    \item Category
    \item Status with color coding
    \item Submission date
    \item Priority level
\end{itemize}

\paragraph{FR-2.6: View Complaint Details}
The system shall allow users to view complete details of their complaints including:
\begin{itemize}
    \item All submission information
    \item Current status and assigned staff (if applicable)
    \item Status change timeline
    \item Comments and updates
    \item Uploaded images in full resolution
\end{itemize}

\paragraph{FR-2.7: Edit Pending Complaint}
The system shall allow users to edit complaints that have "Open" status within 24 hours of submission.

\paragraph{FR-2.8: Cancel Complaint}
The system shall allow users to cancel their own complaints if the status is "Open".

\paragraph{FR-2.9: Add Comment}
The system shall allow users to add comments to their own complaints at any time.

\paragraph{FR-2.10: Search Complaints}
The system shall allow users to search their complaints by:
\begin{itemize}
    \item Ticket ID
    \item Title keywords
    \item Category
    \item Status
    \item Date range
\end{itemize}

\subsubsection{Administrative Functions}

\paragraph{FR-3.1: View All Complaints}
The system shall allow administrators to view all complaints in the system with filtering options:
\begin{itemize}
    \item By status: All, Open, Assigned, In Progress, Resolved, Closed
    \item By category: All categories or specific category
    \item By priority: All priorities or specific priority
    \item By date range
    \item By assigned staff member
\end{itemize}

\paragraph{FR-3.2: Assign Complaint}
The system shall allow administrators to assign complaints to maintenance staff by:
\begin{itemize}
    \item Selecting available staff members by expertise/category
    \item Setting expected completion date
    \item Adding assignment notes
    \item Automatically updating complaint status to "Assigned"
\end{itemize}

\paragraph{FR-3.3: Reassign Complaint}
The system shall allow administrators to reassign complaints to different staff members with a reason for reassignment.

\paragraph{FR-3.4: Update Priority}
The system shall allow administrators to change the priority level of any complaint.

\paragraph{FR-3.5: Update Status Manually}
The system shall allow administrators to manually update complaint status to any valid state with a comment explaining the change.

\paragraph{FR-3.6: Close Complaint}
The system shall allow administrators to close resolved complaints after verification.

\paragraph{FR-3.7: Reopen Complaint}
The system shall allow administrators to reopen closed complaints if issues persist.

\paragraph{FR-3.8: Delete Complaint}
The system shall allow administrators to delete complaints that are duplicate or spam, with confirmation dialog.

\paragraph{FR-3.9: View Staff Workload}
The system shall provide administrators with a dashboard showing:
\begin{itemize}
    \item Number of complaints assigned to each staff member
    \item Complaints by status for each staff member
    \item Average resolution time per staff member
    \item Staff availability status
\end{itemize}

\paragraph{FR-3.10: Manage Users}
The system shall allow administrators to:

\begin{itemize}

\paragraph{FR-4.4: Upload Completion Photos}

\paragraph{FR-4.5: Request Reassignment}

\paragraph{FR-4.6: View Complaint History}
The system shall allow maintenance staff to view their complaint resolution history and performance metrics.

\subsubsection{Notification System}

\paragraph{FR-5.1: Complaint Submission Notification}

\paragraph{FR-5.2: Assignment Notification}

\begin{itemize}

\paragraph{FR-5.4: Comment Notification}

\section{Glossary of Terms}

\begin{description}
    \item[API] Application Programming Interface - A set of protocols for building software applications
    \item[Assignment] The act of allocating a complaint to a specific maintenance staff member
    \item[Audit Trail] A chronological record of system activities
    \item[Backend] Server-side application logic and database operations
    \item[Category] Classification of complaint type (Electrical, Plumbing, IT, etc.)
    \item[Cloudinary] Third-party cloud service for image storage and delivery
    \item[Comment] A textual note added to a complaint by users or staff
    \item[CRUD] Create, Read, Update, Delete - Basic database operations
    \item[Dashboard] Main interface showing summary information and statistics
    \item[End User] Students, faculty, or staff who submit complaints
    \item[Express.js] Web application framework for Node.js
    \item[Frontend] Client-side user interface of the application
    \item[HTTPS] Secure version of HTTP protocol with encryption
    \item[JWT] JSON Web Token - Compact, URL-safe token for authentication
    \item[Kanban] Agile project management methodology using visual workflow
    \item[MERN Stack] Technology stack using MongoDB, Express, React, and Node.js
    \item[MongoDB] NoSQL document database
    \item[Next.js] React framework for production-grade applications
    \item[Node.js] JavaScript runtime environment for server-side programming
    \item[ODM] Object Document Mapper - Maps objects to database documents
    \item[Priority] Urgency level of a complaint (Low, Medium, High, Critical)
    \item[React] JavaScript library for building user interfaces
    \item[REST] Representational State Transfer - Architectural style for APIs
    \item[Role] User permission level (End User, Staff, Administrator)
    \item[Status] Current state of complaint (Open, Assigned, In Progress, Resolved, Closed)
    \item[Ticket ID] Unique identifier for each complaint
    \item[Workload] Number of complaints assigned to a staff member
    \item[Work Queue] List of complaints assigned to maintenance staff
\end{description}
